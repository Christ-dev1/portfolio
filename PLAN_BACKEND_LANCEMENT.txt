1. Diagnostiquer le lancement du backend

a. On vérifie le framework et la configuration avec php artisan about.
Pourquoi : cela montre que le backend est un projet Laravel et quelle base de données Laravel essaie d'utiliser.

b. On teste les migrations avec php artisan migrate:status.
Pourquoi : si la base de données bloque, cette commande affiche l'erreur avant même de lancer le serveur.

2. Corriger la configuration locale

a. On passe DB_CONNECTION de mysql à sqlite dans portfolio-backend/.env.
Pourquoi : le fichier .env.example du projet est prévu pour SQLite et c'est plus simple pour démarrer en local.

b. On définit DB_DATABASE sur database/database.sqlite.
Pourquoi : Laravel doit savoir dans quel fichier stocker la base SQLite.

c. On génère APP_KEY avec php artisan key:generate.
Pourquoi : Laravel a besoin d'une clé d'application pour fonctionner correctement.

3. Préparer la base de données

a. On crée le fichier database/database.sqlite.
Pourquoi : SQLite utilise un fichier local comme base de données.

b. On lance php artisan migrate.
Pourquoi : cela crée les tables users, cache, jobs, projects, blog_posts et messages.

4. Lancer le serveur

a. On lance php artisan serve.
Pourquoi : c'est la commande Laravel qui démarre le backend en local.
