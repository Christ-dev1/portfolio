PLAN POUR CREER LE BACKEND LARAVEL DU PORTFOLIO

Objectif general :
Creer un backend Laravel simple pour gerer les projets du portfolio depuis une interface admin, afficher des KPI utiles, analyser les messages/visites du site, et recevoir par email les messages envoyes depuis le formulaire de contact.

Important :
On ne va pas tout coder en une seule fois.
On avance etape par etape pour que tu voies l'evolution du projet et que tu comprennes ce qu'on ajoute.


1. Creer le projet Laravel

a. On va creer le dossier backend avec la commande :
composer create-project laravel/laravel portfolio-backend

Pourquoi :
Laravel va creer une structure propre avec les dossiers app, routes, database, config, resources et public.
Ce sera la base de notre backend.

b. On va entrer dans le dossier backend :
cd portfolio-backend

c. On va lancer le serveur Laravel :
php artisan serve

Pourquoi :
Cela permet de verifier que Laravel fonctionne avant d'ajouter du code.


2. Configurer la base de donnees

a. Pour commencer simplement, on va utiliser SQLite.
On va creer le fichier :
database/database.sqlite

b. Dans le fichier .env, on va configurer :
DB_CONNECTION=sqlite
DB_DATABASE=database/database.sqlite

Pourquoi :
SQLite est plus simple pour apprendre, car il n'y a pas besoin d'installer MySQL au debut.

c. On va lancer :
php artisan migrate

Pourquoi :
Laravel va creer les tables de base utiles au fonctionnement du projet.


3. Creer l'authentification admin

a. On va installer Laravel Breeze :
composer require laravel/breeze --dev

b. On va generer l'authentification :
php artisan breeze:install blade

c. On va installer les dependances front de Laravel :
npm install
npm run build

d. On va lancer les migrations :
php artisan migrate

Pourquoi :
Breeze donne un systeme simple de connexion, inscription et deconnexion.
Pour ton projet, l'espace admin sera protege par la connexion.

Remarque :
Au debut, on gardera un seul type d'utilisateur : admin.
On evitera les roles compliques tant que la base n'est pas claire.


4. Creer la gestion des projets

a. On va creer le modele Project avec migration, controller et factory :
php artisan make:model Project -mcf

b. Dans la migration, on va preparer les colonnes :
- title : titre du projet
- slug : lien lisible du projet
- description : description courte
- content : description longue
- image : chemin de l'image
- github_url : lien GitHub
- demo_url : lien de demonstration
- technologies : liste des technologies
- is_featured : projet mis en avant
- published_at : date de publication

Pourquoi :
Ces champs permettent d'afficher les projets dans ton portfolio et de les gerer depuis l'admin.

c. On va creer les routes admin pour ajouter, modifier, supprimer et lister les projets.

d. On va creer les vues Blade :
- resources/views/admin/projects/index.blade.php
- resources/views/admin/projects/create.blade.php
- resources/views/admin/projects/edit.blade.php
- resources/views/admin/projects/show.blade.php

Pourquoi :
Ces pages seront l'interface admin simple pour gerer les projets sans utiliser la base de donnees directement.


5. Creer l'API publique pour le front Next.js

a. On va creer une route API :
GET /api/projects

Pourquoi :
Le front-end Next.js pourra recuperer les projets depuis Laravel.

b. On va creer une route pour un seul projet :
GET /api/projects/{slug}

Pourquoi :
Cela permettra plus tard d'afficher une page detail d'un projet.

c. On va tester ces routes avec le navigateur ou Postman.


6. Relier le front-end aux projets du backend

a. Dans front-end/.env.local, on va ajouter :
NEXT_PUBLIC_API_URL=http://127.0.0.1:8000/api

b. Dans le composant Projects.tsx, on remplacera les projets ecrits en dur par un appel API vers Laravel.

Pourquoi :
Quand tu ajouteras un projet dans l'admin Laravel, il pourra apparaitre dans le portfolio sans modifier le code du front.


7. Creer le formulaire de contact cote backend

a. On va creer le modele ContactMessage :
php artisan make:model ContactMessage -m

b. Dans la migration, on va ajouter :
- name
- email
- subject
- message
- ip_address
- user_agent
- read_at

Pourquoi :
On garde une trace des messages recus, meme si l'email ne part pas ou si on veut les consulter dans l'admin.

c. On va creer le controller :
php artisan make:controller Api/ContactController

d. On va creer la route :
POST /api/contact

Pourquoi :
Le formulaire du front enverra les donnees vers cette route.


8. Envoyer l'email quand un message est recu

a. On va creer un mail Laravel :
php artisan make:mail ContactMessageMail

b. Dans le fichier .env, on configurera les variables mail :
MAIL_MAILER=smtp
MAIL_HOST=
MAIL_PORT=
MAIL_USERNAME=
MAIL_PASSWORD=
MAIL_FROM_ADDRESS=
ADMIN_EMAIL=

Pourquoi :
Laravel utilisera ces informations pour envoyer le message vers ton adresse email.

c. Dans ContactController, apres validation du formulaire, on enregistrera le message puis on enverra l'email a ADMIN_EMAIL.

Remarque :
Pour apprendre proprement, on testera d'abord avec Mailtrap ou un service SMTP de test.
Ensuite seulement, on mettra ton vrai email.


9. Creer l'admin pour consulter les messages

a. On va creer un controller admin :
php artisan make:controller Admin/ContactMessageController

b. On va creer les pages :
- liste des messages
- detail d'un message
- marquer comme lu
- supprimer un message

Pourquoi :
Tu pourras consulter les contacts depuis l'admin, pas seulement par email.


10. Ajouter les KPI admin

a. On va creer une page dashboard :
resources/views/admin/dashboard.blade.php

b. On va afficher des KPI simples :
- nombre total de projets
- nombre de projets publies
- nombre de messages recus
- nombre de messages non lus
- dernier message recu
- nombre de projets mis en avant

Pourquoi :
Ces indicateurs donnent une vision rapide de l'etat du portfolio.

c. Plus tard, on pourra ajouter des KPI plus avances :
- nombre de visites
- pages les plus vues
- projets les plus consultes
- taux de conversion contact
- evolution des messages par mois


11. Ajouter une analyse des visites du site

a. On va creer le modele SiteVisit :
php artisan make:model SiteVisit -m

b. On va stocker :
- page_url
- referrer
- ip_address
- user_agent
- visited_at

c. On va creer une route API :
POST /api/visits

Pourquoi :
Le front pourra envoyer une visite quand un utilisateur arrive sur une page.
Cela permettra de construire des statistiques simples.

d. Dans le dashboard admin, on affichera :
- visites totales
- visites aujourd'hui
- pages les plus visitees
- dernieres visites


12. Securiser progressivement

a. On va proteger les routes admin avec le middleware auth.

b. On va valider toutes les donnees envoyees par les formulaires.

c. On va limiter le formulaire de contact pour eviter trop d'envois rapides.

d. On va proteger les routes API avec CORS correctement configure.

Pourquoi :
Meme pour un projet academique, il faut apprendre les bons reflexes de securite.


13. Tester chaque partie

a. Apres chaque grande etape, on testera :
- le serveur Laravel
- les migrations
- la connexion admin
- l'ajout d'un projet
- la modification d'un projet
- la suppression d'un projet
- l'affichage des projets dans l'API
- l'envoi du formulaire contact
- la reception de l'email
- l'affichage des KPI

Pourquoi :
Tester petit a petit evite de se retrouver avec beaucoup d'erreurs difficiles a comprendre.


14. Ordre conseille pour avancer

Etape 1 :
Creer le projet Laravel et le lancer.

Etape 2 :
Configurer SQLite et les migrations.

Etape 3 :
Installer l'authentification admin.

Etape 4 :
Creer le CRUD des projets.

Etape 5 :
Creer l'API publique des projets.

Etape 6 :
Connecter le front Next.js a l'API.

Etape 7 :
Creer le formulaire contact cote Laravel.

Etape 8 :
Envoyer les emails.

Etape 9 :
Creer le dashboard KPI.

Etape 10 :
Ajouter les statistiques de visites.


Prochaine action proposee :
Commencer par l'etape 1 seulement :
composer create-project laravel/laravel portfolio-backend
